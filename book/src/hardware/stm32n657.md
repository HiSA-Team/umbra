# STM32N657 — NUCLEO-N657X0-Q

## Board Overview

The NUCLEO-N657X0-Q hosts an STM32N657X0 — a Cortex-M55 (ARMv8.1-M)
with TrustZone-M, hardware crypto, an integrated NPU (ST Neural-ART
Accelerator), and **no internal flash**. The MCU boots from external
XSPI2 via the on-board MX25UM51245G (512 Mb) NOR. Umbra runs as the
first-stage bootloader (FSBL) loaded by the Boot ROM into AXISRAM2.

## Key Characteristics

| Feature | Detail |
|---|---|
| MCU | STM32N657X0H3Q |
| Core | Cortex-M55, 800 MHz (CPU) / 400 MHz (AXI) / 200 MHz (HCLK) |
| Internal flash | **None** — FSBL lives in external XSPI2 NOR |
| SRAM | ~4.2 MB total (FLEXRAM, AXISRAM1–6, AHBSRAM1–2, plus 256 KB ITCM and 256 KB DTCM) |
| External flash | MX25UM51245G 64 MB / 512 Mb XSPI2 (memory-mapped at `0x70000000`) |
| HW AES | CRYP1 (fast AES, 11 cycles/block) + SAES (DPA-resistant) |
| HW HMAC | HASH peripheral (SHA-1/224/256/384/512, HMAC); chained measurement runs in HW (~9 s for 11 MB) |
| External flash decryption | MCE1–MCE4 (currently passthrough on this board — see boot crate's `oracle.rs`) |
| NPU | ST Neural-ART, 900 MHz (clocked via PLL3 + IC6) |
| Debug UART | USART1 (PE5 TX / PE6 RX, AF7) via ST-Link VCP, **115200 baud** |
| LEDs | PG0 (green), PG8 (blue), PG10 (red) |
| Boot jumper | JP2 — Dev-Boot (2-3) for flashing, Flash-Boot (1-2) for run |
| Enclave storage | XSPI2 (header at `0x70090000`) |
| Block loading | CPU copy via memory-mapped XSPI2 read (MCE2 in passthrough) |

## UART Connection

The ST-Link provides a Virtual COM Port. No additional wiring needed.

- **Serial device** (macOS): `/dev/cu.usbmodem*`
- **Serial device** (Linux): `/dev/ttyACM0`
- **Baud rate**: 115200

## IDAU Aliasing

The N6 uses a different aliasing scheme from STM32L5: **the top nibble
encodes security state**.

| Top nibble | Security | Example |
|---|---|---|
| `0x0`, `0x2`, `0x4` | Non-Secure | `0x24000000` AXISRAM1 NS view |
| `0x1`, `0x3`, `0x5` | Secure / NSC | `0x34000000` AXISRAM1 Secure view |

SAU can re-classify NS regions in the even-nibble alias range. Secure
code always uses the odd-nibble alias.

## Memory Firewall (RIF)

The N6 replaces L5's GTZC with the RIF family:

| Block | Purpose | Key registers |
|---|---|---|
| RISAF1..23 | Per-memory access control (start/end addr, S/NS, R/W, CID) | RISAF`x`->CFGR, REGCFGR, BASEREG, ENDREG |
| RIFSC | Per-peripheral CID-aware enable | SECCFGR`x`, PRIVCFGR`x`, RCFGLOCKR |
| RIMC | Bus master CID tagging (DMA, NPU) | RIMC_MASTERCFG_*, RIMC_ATTR |
| IAC | Central illegal-access reporting | IAC_ISR, IAC_IER |

Umbra programs these in `PlatformBoot::init_security`. The Phase E.1
bring-up landmine to watch for: the IDAU "AXISRAM1 view" at
`0x34000000` is split across **RISAF7** (FLEXRAM, offsets < 0x64000)
and **RISAF2** (proper AXISRAM1, offsets ≥ 0x64000). Configuring only
RISAF2 leaves a hole.

## Flash Workflow

1. Set JP2 = **Dev-Boot** (2-3).
2. Plug the board in (CN1, ST-Link USB).
3. Build:
   ```bash
   export MCU_VARIANT=stm32n657
   source ./settings.sh
   ./rebuild_all.sh
   ```
4. Flash:
   ```bash
   tools/flash_n657.sh
   ```
5. Set JP2 = **Flash-Boot** (1-2) and reset.

`flash_n657.sh` writes:
- FSBL signed image at XSPI2 `0x70000000`
- Host bin at XSPI2 `0x70080000`
- (Object-detection demo) model bytecode and `network_data.xSPI2.bin`
  further up

## Debug

Boot ROM holds the core until authentication and copy are done, so a
clean SWD reset is not possible. Attach without reset:

```bash
openocd -f openocd_scripts/stm32n6x.cfg
arm-none-eabi-gdb path/to/boot \
    -ex 'target extended-remote :3333'
```

For NPU work, the activations live in AXISRAM5/6 — those pages are
power-gated by default and `init_clocks` opens them as part of the
NPU bring-up (`G.1.a.3`).
